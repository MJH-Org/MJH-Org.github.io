2.1 请分别说明C/S、B/S的三点主要区别
部署和安装：B/S程序不需要在用户的机器上安装专门的客户端软件；
用户界面和交互：C/S客户端应用程序通常提供丰富的用户界面和交互能力，B/S用户通过网页浏览器与应用程序进行交互；
网络通信和带宽要求，C/S的客户端与服务器之间的交互可能会更加频繁和复杂，C/S架构可能对网络质量和带宽有更高的要求，B/S架构的应用程序通常通过HTTP协议进行状态无关的交互，这可能减少了对网络的要求；
跨平台兼容性方面，B/S更好；
数据安全性：B/S更好；
维护和升级：B/S更好；
资源利用：C/S可以更好利用本地资源，B/S主要利用服务器资源；
离线工作能力：C/S的应用程序可能更容易支持离线工作。
2.2 请分别说明SaaS、PaaS、IaaS三种服务模式的中文含义和相关特点。
SaaS是指软件即服务，特点是消费者使用应用程序，但并不掌握操作系统、硬件或运作的网络基础架构，它以服务观念为基础，软件服务供应商以租赁的概念提供客户服务，而非购买，比较常见的模式是提供一组账号密码。
PaaS是指平台即服务，特点是消费者使用主机操作应用程序，消费者掌控运作应用程序的环境（也拥有主机部分掌握权），但并不掌握操作系统、硬件或运作的网络基础架构。
IaaS是指基础设施即服务，特点是消费者使用“基础计算资源”，如处理能力、存储空间、网络组件或中间件，消费者能掌握操作系统、存储空间、已部署的应用程序及网络组件（如防火墙、负载平衡等），但并不掌控基础架构。
2.3 规划会计信息系统时需要进行可行性分析，请说明可行性分析需要考虑的技术可行性、经济可行性和管理可行性三个方面的内容。
一是技术可行性。是指根据现有的技术条件，能否达到所提出的要求；所需要的物理资源是否具备，能否得到。技术条件包括硬件、系统软件、应用软件和技术人员。
二是经济可行性。要估计系统的成本和效益，分析系统经济上是否合理。如果不能提供开发系统所需要的资金，或者不能够提高企业的效益，就不应该开发此系统。所以，经济可行性需要解决两个问题：资金可行性和经济合理性。
三是管理可行性。考虑所建立的系统能否在该企业实现，在当前操作环境下能否很好的运行，即组织内外是否具备接受和使用新系统的条件。从组织内部讲，新系统的建立可能导致某些制度甚至管理体制的变动，组织的能力影响系统的生存。从组织外部讲，新系统运行后，报表、格式的改变是否被有关部门认可和接受，将直接影响企业的经营。
2.4 简述总账子系统与其他子系统之间的关系。

2.6 请说明在会计信息系统中，常用的三种编码方法的含义和特点。
一是顺序编码。顺序编码以连续的数字按升序或降序排列。特点是简便易行，但扩展性较差。
二是分组编码。用若干连续的数据组成几个系列编码对象的编码。这种编码方法系列性、扩展性好，但空码较多。
三是位数编码。以分类对象的从属层次关系为排列顺序的编码。编码分为多层，编码左端为高层次编码，右端为低层次编码，每一层次可按顺序编码或分组编码进行编码。这种编码在会计信息系统中具有广泛的应用，编码结构简单，易于扩展，易于分类汇总。

2.10 系统规划 、系统分析 、系统设计、系统实施等各阶段的主要任务及产生的主要文件，并说明文件的内容。
（一）系统规划阶段
系统规划阶段的主要任务是对企业的环境、目标、现行系统的状况进行初步调查，明确现行系统存在的问题，并根据企业目标和发展战略，对建立新系统的需求做出分析和预测。此阶段可能产生的文件主要包括：
可行性项目建议书：该文件详细阐述了建立新系统的可行性分析，包括技术可行性、经济可行性、操作可行性等方面的内容，是系统规划阶段的重要输出。
（二）系统分析阶段
系统分析阶段（逻辑设计阶段）的主要任务是对现行系统进行详细调查和全面分析，描述现行系统的业务流程，指出现行系统的局限性与不足之处，并确定新系统的目标和逻辑功能要求。此阶段可能产生的文件主要包括：
系统分析报告：该文件详细描述了新系统的逻辑模型，包括系统的业务流程、数据流程、功能需求等方面的内容，是系统分析阶段的核心输出。
系统说明书：该文件在系统分析阶段产生，用于详细描述系统的逻辑功能需求，包括系统的输入输出、处理逻辑、数据流等关键信息。
（三）系统设计阶段
系统设计阶段的主要任务是根据系统分析报告中规定的功能，结合实际条件，具体设计实现逻辑模型的技术方案。此阶段可能产生的文件较多，主要包括：
需求文档：在系统设计前收集的用户需求的文档，其中包含了用户对系统功能和性能的具体要求。
功能需求规格说明书：详细定义了系统的功能需求，包括系统的各个模块和子系统的功能描述、输入输出、处理逻辑等。
数据模型文档：描述了系统中所使用的数据模型，包括实体、属性和关系等，通常使用E-R图或UML类图来表示。
系统设计文档：描述了系统的整体结构和组成部分，包括模块划分、模块间的接口定义、流程图、数据流图等。
界面设计文档：描述了系统的用户界面设计，包括页面布局、交互方式、按钮和输入框的位置等。
数据库设计文档：详细定义了系统所使用的数据库结构，包括表的定义、字段的定义、索引、关系等。
系统架构文档：描述了系统的整体架构，包括硬件平台、网络拓扑、系统模块划分等。
（四）系统实施阶段
系统实施阶段的主要任务是按照系统设计文档进行系统的建设、安装、调试和部署，确保系统能够正常运行并满足用户需求。此阶段可能产生的文件主要包括：
系统实施方案：详细描述了系统实施的具体步骤、时间计划、资源分配等方面的内容，是系统实施阶段的重要指导文件。
系统测试报告：在系统实施阶段进行各种测试（如单元测试、集成测试、系统测试等）后产生的报告，用于记录测试结果、分析测试中发现的问题并提出解决方案。
用户手册：为系统的最终用户提供了使用指导和操作说明，帮助用户熟悉和掌握系统的使用方法。
部署手册：描述了系统的部署和安装过程，以及配置要求和环境设置等，为系统管理员或运维人员提供了详细的操作指南。
2.11 请说明业财融合的含义、内容、主要工作、主要意义。
（一）业财融合的含义
业财融合，是业务与财务融合的简称，是指业务发展与财务管理相结合，业务和财务融为一体，从企业的整体去思考业务开展是否符合集团发展的目标方向。它强调将传统上独立的业务和财务部门整合为一个协同工作的整体，通过更紧密的合作和信息流动，使企业能够更灵活、迅速地应对市场变化，实现业务增长和管理效率的提升。
（二）业财融合的内容
财务管理业务和财务核算业务的融合：财务管理业务不仅仅是会计核算，还包括资金管理、税务管理、资产管理、预算管理等。业财融合也不仅仅是其他业务部门和财务核算的融合，财务管理业务和财务核算业务的融合更是业财融合的起点和基础。
业务管理数据和财务核算业务的融合：通过信息化手段，将业务管理数据同步至财务核算系统，实现业务与财务数据的无缝对接。
财务管理数据和业务管理数据的融合：财务部门通过深入分析财务管理数据，为业务部门提供有价值的决策支持，同时业务部门也将实时业务数据反馈给财务部门，以便财务部门更好地进行财务规划和预测。
企业外部数据和企业内部数据的融合：企业不仅需要关注内部数据，还需要关注外部数据，如市场趋势、竞争对手动态等。通过将这些外部数据与内部数据相结合，企业可以做出更明智的决策。
（三）业财融合还包括以下几个方面的工作：
事前规划：业务部门在了解项目情况后，结合市场、竞品和客户开发情况，作出合理的预计。财务部门则根据这些预计进行预算制定和资金筹备。
事中控制：在业务实际开展过程中，财务部门实时监控销售进度、成本开支和项目利润情况，并与业务部门共同分析存在的问题及改进措施。同时，业务部门也需要关注财务风险，及时识别并规避风险。
事后评估：在项目结束后，财务部门对整个项目的实际数据与预算数据进行对比分析，为项目相关人员的绩效评价和绩效考核提供依据，并总结项目整体管理中的经验及教训，为新项目的开展提供改进意见。
（四）业财融合的意义
业财融合的意义在于消除业务和财务之间的隔阂，使财务能够更深度地参与业务决策和战略制定，同时业务侧也更加理解和借助财务数据进行更科学、有效的经营。这有助于提升企业的运营效率和管理水平，实现企业的可持续发展。
2.12 REA模型概念、要素与核心关系
一、REA 模型定义
REA 模型全称 资源 — 事件 — 参与者（Resource-Event-Agent），是 1982 年由美国学者 McCarthy 提出的会计信息系统概念数据模型。它跳出传统会计科目与借贷分录框架，直接以企业真实业务实体建模，按资源、业务事件、参与主体及其关系设计数据库。
二、三大核心要素
R 资源（Resource）：企业拥有或控制、具有经济价值、可计量的资产。例如：现金、银行存款、存货、固定资产、原材料等。
E 事件（Event）：引起资源增减变动的经营业务活动。例如：采购入库、销售出库、收款、付款、生产领料、完工入库等。
A 参与者（Agent）：参与业务事件的内外部个人或组织。例如，内部：采购员、销售员、仓库管理员、财务人员；外部：供应商、客户、银行。
三、REA 核心关系
经济对偶关系（Give-Get）：任何业务都是付出资源、获取资源的对等交换。例如：采购 → 付出现金、获得存货；销售 → 付出存货、收到现金。
事件与资源关联：业务事件触发资源增加或减少。
事件与参与者关联：每个业务都有对应的内部、外部参与者。

2.14 ERP的概念及与会计信息系统的区别是什么？
一、ERP 概念
ERP（Enterprise Resource Planning）企业资源计划，是在 MRPⅡ 基础上发展起来的企业一体化集成管理信息系统。它把企业采购、销售、生产、库存、财务、人力、资产等所有资源进行统一规划、管控与共享，实现业务、财务、管理一体化，优化资源配置，提升企业整体运营与决策效率。
二、会计信息系统（AIS）概念
会计信息系统是以会计核算为核心，专门处理账务、报表、应收应付、存货、薪资、固定资产等会计业务，完成会计数据采集、加工、存储、输出，主要服务财务核算与简单管理。
三、ERP 与 会计信息系统的区别（重点背诵）
范围不同：AIS 只局限财务会计领域；ERP 覆盖全企业业务 + 财务 + 人力 + 生产 + 供应链所有资源。
层级不同：AIS 属于部门级系统，只为财务部门服务；ERP 属于企业级整体系统，面向全公司各部门。
核心定位不同：AIS 以会计核算为主；ERP 以企业资源整体规划、流程管控、经营决策为主。
数据关系不同：传统 AIS 业务与财务分离、数据孤岛；ERP 实现业财一体化，业务自动生成财务凭证，数据实时共享。
功能广度不同：AIS 只有账务、报表、往来、薪资、固定资产等财务功能；ERP 包含供应链、生产制造、人力、预算、成本、决策分析等全方位管理。
管理深度不同AIS 侧重事后核算；ERP 实现事前预算、事中控制、事后分析全过程管理。

3.1 请说明该系统报表系统的操作与类似excel这样的电子表格系统的操作的区别。
格式设计方面，该系统主要满足报表的格式设计要求，没有excel强大和方便。
数据引用方面，该系统设计了专门的接口可以直接引用系统内的数据，而且该系统不允许直接填写数据，比较规范，不容易出错，而excel需要直接填写数据或依靠第三方系统提供数据接口。
数据计算方面，该系统只有有限的数据计算功能，Excel拥有强大的数据计算功能。
编程方面，该系统没有提供编程语言和接口，而excel可以使用VBA编程或使用第三方语言与excel协同工作。
3.2 请说明使用手工做账与使用用友U8系统做账有哪些区别，两者的优缺点各是什么。
流程上主要有五点不同：数据处理的起点与终点不同；数据处理方式不同；数据存储方式不同；对账方式不同；查询方式不同。
该系统的优点：优点主要体现在，流程上有很多工作可以自动化。比如，记账都是根据原始凭证记账，但该系统某些账务可以通过记账凭证的模板自动记账；登账在该系统中是自动完成；可以根据设计的报表自动生成报表。
手工的优点：数据更容易持久保存，而电子的需要额外采用数据保存的方法；手工在某些小公司，可能成本更低。
手工的缺点：数据大量重复，工作强度大；信息提供不及时，准确性差。

集成工作流、电子档案、移动应用；
采用角色权限、日志安全管控。
3.8 简述用友 U8 完整发展历程。
第一阶段：DOS 财务软件时代（早期）：早期用友以单机 DOS 版财务软件为主，只有简单账务、报表，纯会计核算，无业务模块，Access 小型数据库，只能手工记账替代。
第二阶段：Windows 财务转型期（1998 年前）。升级为Windows 版本，界面可视化，开始具备总账、报表、固定资产、薪资等基础模块，仍以财务单一系统为主，未集成供应链。
第三阶段：U8 诞生 财务 + 业务集成（1998—2009）：经典版本：U861 → U872 → U890。支持 Web 远程、SQL 大型数据库，引入集团合并报表。加入生产制造、供应链、HR，正式从财务软件转向ERP。功能完善，模块齐全，成为国内中小企业主流 ERP。
第四阶段：U8+ 云化转型期（2010—2016）：U8V10、V11、V12、V13。特点：升级三层 C/S + B/S 架构。支持网络集中部署、角色权限、工作流、电子发票；实现业财一体化、集团集中管控。
第五阶段：互联网与云时代（2017 至今）：推出 U8 Cloud、U8+V16/V18。全面上云，支持智能制造、电子档案、RPA 机器人、移动端应用。从传统套装软件转向云 ERP服务模式。

4.1 ERPNext核心架构中用到的主要软件和技术组件有哪些？
ERPNext 是一个基于 Python 和 JavaScript 开发的开源企业资源规划系统。它并非单一的软件，而是由一系列开源技术和框架组成的完整技术栈。
以下是 ERPNext 核心架构中用到的主要软件和技术组件：
核心开发框架与语言
Frappe Framework: 这是 ERPNext 的基石。Frappe 是一个全栈的 Web 应用框架，用于构建数据库模型、处理 API 请求、管理用户权限和生成前端界面。
Python: 后端主要编程语言，用于处理业务逻辑、后台任务和数据库操作。
JavaScript (jQuery/Vue.js): 前端主要编程语言，用于实现客户端逻辑、表单交互和动态界面。
数据库与缓存
MariaDB (或 MySQL): 默认的关系型数据库，用于存储所有业务数据（如订单、发票、库存等）。
Redis: 内存数据结构存储，用作缓存系统，用于加速数据读取、管理会话以及处理后台任务队列。
服务器与 Web 服务
Nginx: 用作反向代理服务器和 Web 服务器，负责处理静态文件服务和负载均衡。
Gunicorn: WSGI HTTP 服务器，用于运行 Python 应用程序代码。
Node.js: 用于构建前端资源、Socket.IO 实时通信以及运行某些服务端 JavaScript 脚本。
操作系统与部署
Linux: 推荐且最常用的部署环境（如 Ubuntu, Debian, CentOS）。
Docker: 官方提供了基于 Docker 的部署方案，用于容器化部署，简化环境配置。
4.2 ERPNext中进行账务处理的实验里：期初开帐时，“库存商品”与其他科目在录入方式上有何不同？为什么要引入“临时开账科目”？
参考答案
录入方式不同： 其他科目直接通过“日记账凭证”录入期初余额；而“库存商品”不能通过日记账凭证录入，必须在“库存”模块下的“库存调账”中增加，目的选择“期初库存”。
引入原因： 在录入除库存商品外的其他科目时，借贷方会因为缺少库存商品的价值而不平衡。引入“临时开账科目”作为过渡科目，可以使期初借贷方强制平衡，待后续通过库存调账补录库存商品后，该科目余额会自动调整。
4.3 ERPNext中进行账务处理的实验里：三单一致指什么？目的是什么？在EPRNext中如何体现（以销售为例）？
参考答案：
在 ERPNext 的财务共享与业务流程中，是指通过系统强制关联 订单（Order）、货单（Delivery/Receipt） 和 发票（Invoice），以确保业务流、资金流和票据流的一致性。
在 ERPNext 的做账实验中，“三单”具体指：
订单（Order）： 业务的起点，代表双方达成的交易意向。
销售场景：销售订单（Sales Order）
采购场景：采购订单（Purchase Order）
货单（Goods Receipt/Delivery）： 实物的流转凭证，代表货物的实际进出。
销售场景：发货单（Delivery Note）
采购场景：入库单（Purchase Receipt）
发票（Invoice）： 财务结算凭证，代表应收或应付的款项。
销售场景：销售发票（Sales Invoice）
采购场景：采购发票（Purchase Invoice）
文档中明确指出，“三单一致”的目的是：
业务与财务融合： 确保财务记账（生成会计凭证）是基于真实的业务单据发生的，而不是凭空录入分录。
状态追踪： 通过单据之间的关联，系统可以自动更新状态（如“未发货未付款”、“已发货未收款”、“已完成”），从而提醒会计人员跟进后续的做账或核销工作。
税务合规： 确保发票开具的数量和金额与实际发货/收货的数量一致，符合税务监管要求。
在 ERPNext 中如何实现“一致”？以销售为例
先录单： 先录入“销售订单”（Sales Order），记录客户要买什么、多少钱。
再发货： 在销售订单页面点击“新建” -> 发货单（Delivery Note），记录货物实际发出了。
后开票： 在发货单页面点击“新建” -> 销售发票（Sales Invoice），根据实际发出的货物开具发票。
结果： 发票金额 = 订单金额 = 货物价值。此时，系统自动生成“主营业务收入”和“应收账款”分录。
总结：在做账实验中，“三单一致”意味着不能直接手工录入会计凭证（Journal Entry）来记录进销存业务，而是必须通过上述的业务单据流转，让 ERPNext 系统自动根据“发票”生成会计凭证。这样做的好处是，每一笔账都有对应的业务源头（订单和货单）作为审计依据。
4.4 ERPNext中进行账务处理的实验里：应收应付的核销如何实现？
参考答案
在 ERPNext 中，应收（Accounts Receivable）与应付（Accounts Payable）的核销，本质上是将业务单据（发票）与资金单据（银行流水）进行勾稽匹配的过程。
这一过程主要通过**“收付款凭证（Payment Entry）”与“银行流水（Bank Transaction）”的核销（Reconciliation）**功能来实现。在 ERPNext 中，核销不仅仅是财务记账，而是业务流与资金流的对齐：
业务侧（债权/债务）： 由销售/采购发票（Sales/Purchase Invoice）产生应收/应付。
资金侧（实收/实付）： 由银行账户的实际进出账（Bank Transaction）记录资金流动。
核销动作： 通过“收付款凭证”作为桥梁，连接发票与银行流水。
在系统中如何体现？核销完成后，系统会在以下几个维度发生变化：
（1）单据状态：核销前的状态，收付款凭证状态为“未提交”或“已提交”；银行流水状态为“未核销”。核销后的状态，收付款凭证状态变为“已核销”；银行流水状态变为**“已核销（Reconciled）”**。
（2）余额影响：核销前的状态，应收账款/应付账款余额未减少；银行存款余额未最终锁定。核销后的状态，应收账款/应付账款余额清零（或减少对应金额）；该笔银行流水被锁定，不再参与后续对账。
（3）报表体现：核销前的状态，试算平衡表中，应收/应付科目有余额。核销后的状态，试算平衡表中，该笔业务的应收/应付余额消失，资金科目余额准确。
（4）特殊情况处理：如滞纳金，直接通过日记账凭证（Journal Entry）记账，然后直接与银行流水核销，不需要经过“收付款凭证”，因为其对应科目不是标准的应收/应付。
总结：在 ERPNext 中，应收应付的核销 = 收付款凭证（关联发票） + 银行流水（实际资金）的匹配。这一过程确保了文档中提到的**“做到应收应付与银行实际的收付一致”**，是验证 FSSC（财务共享服务中心）做账准确性的核心环节。
